AI判斷邏輯設計

一、設計目的
本文件針對本專題中 AI Agent 之「情境判斷核心模組」進行設計說明。
本模組負責在 AI 持續監看聊天室對話的過程中，
將對話轉換為摘要後，判斷目前情境並對應至適當劇本，
作為後續行為觸發之依據。

二、模組位置
本模組位於 AI Agent 整體流程圖之「情境判斷」階段。
系統流程如下：
對話接收 → 即時摘要 → 情境判斷 → 行為觸發 → 介入控制

三、輸入與輸出定義
輸入：即時摘要資料（由摘要模組產生）
輸出：劇本名稱（17種情境之一）

補充：輸出格式（標準化）
劇本名稱：戶外活動與天氣影響
判斷依據：activity_info = 戶外，risk_info = 天氣
建議行為：提供室內替代方案
confidence_score：0.85

四、摘要欄位設計（判斷依據）
time_info：時間資訊
location_info：地點資訊
activity_info：活動類型
people_count：人數
budget_info：預算
options：候選選項
decision_state：討論狀態
risk_info：風險
need_type：需求

即時摘要生成方式：
本系統以前 N 則群組訊息作為滑動對話視窗（sliding window），
隨著新訊息進入持續更新摘要內容，
AI Agent 會先針對該視窗進行摘要，
萃取時間、地點、活動、候選選項、討論狀態與風險資訊，
並轉為結構化資料，作為後續情境判斷之輸入。

【補充定義（供系統判斷使用）】

為確保情境判斷邏輯可被系統正確解析，以下補充部分欄位之判斷標準：

time_info 補充定義：
- 「現在」：對應使用者對話中出現「現在」、「等等」、「馬上」、「剛剛」、「立刻」等即時性語句
- 「固定時間」：對應使用者對話中出現「幾點開始」、「幾點開演」、「幾點集合」、「場次時間」等不可延誤之明確時間點
- 一般時間：如「週六」、「下午」、「晚上」、「明天」等相對時間描述

activity_info 補充定義：
- 「戶外」：對應活動內容包含戶外行為，如「爬山」、「海邊」、「走走」、「逛老街」、「騎腳踏車」、「戶外景點」等
- 「室內」：如餐廳、電影院、展覽館、百貨公司等室內活動
- 「即時需求」：如「現在要不要吃」、「現在去哪」、「附近有什麼」等臨時決策情境

risk_info 補充定義：
- 「時間衝突」：對話中出現時間安排不合理，如活動時間過近、重疊或移動時間不足
- 「天氣」：對話中出現天氣相關疑慮，如「會不會下雨」、「天氣不好」、「颱風」、「太熱／太冷」等

need_type 補充定義：
- 「補資訊」：使用者主動詢問資訊，如價格、營業時間、餐廳內容、交通方式等
- 「決策輔助」：出現猶豫、比較、選擇困難等語句

decision_state 補充補強：
- 「卡住」補充判斷條件：若最近數句對話出現高度重複語意（如「都可以」、「隨便」、「我也可以」），且未新增具體條件或選項，則判定為卡住狀態

五、decision_state 判斷方式
發起中：出現活動或時間，但尚未出現明確選項
卡住：出現「都可以」、「隨便」、重複討論或最近3句無新選項
完成：出現「就這個」、「OK」、「那就這樣」等決策語句

七、判斷優先序
當同時符合多個條件時，依以下順序判斷：
1. 高風險情境（時間衝突、天氣影響）
2. 決策卡住情境（投票、衝突）
3. 行程規劃情境
4. 前期辨識情境
5. 收尾支援情境

補充：
當同時符合多個劇本條件時，
僅選擇優先序最高者作為最終判斷結果，避免重複觸發。

八、Fallback機制
若無任何條件符合，
則預設判斷為「前期辨識情境」，
並持續觀察後續對話進行重新判斷。

九、AI信心值
每次判斷皆輸出 confidence_score（0～1）
0.8～1.0：高度確定
0.5～0.8：中等確定
0.5以下：低確定（建議持續觀察）

十、情境判斷流程
1. 判斷是否為高風險情境
2. 判斷是否為決策卡住或多選項
3. 判斷是否可生成完整行程
4. 若條件不足 → 前期辨識
5. 若已完成 → 收尾支援

當 AI 完成情境判斷後，
系統將依據劇本名稱啟動對應後續處理模組，
例如推薦模組、投票模組、風險提醒模組、
行程生成模組或執行支援模組。
